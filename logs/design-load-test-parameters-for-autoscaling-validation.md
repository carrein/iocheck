Good question — these values aren't arbitrary, they need to satisfy a specific story arc. Let me walk through the math, then propose concrete numbers.

## The story we're engineering

For the demo to land, we need a specific regime:

1. **CPU stays well below 70% of request** under burst — otherwise CPU HPA fires and our "CPU is wrong" thesis collapses.
2. **Latency degrades visibly during burst** — driven by something *other* than CPU. For a Bun/Node service with cache-friendly reads, the real bottleneck is **event-loop queueing + DB pool exhaustion** under concurrency, not raw CPU.
3. **Switching to RPS-based HPA** spreads load across new pods, restoring p99.

So we need to choose CPU request *generously* (so 70% is hard to hit) and shape the workload to be *I/O-bound* (cache-heavy reads).

## How each knob behaves

### CPU per request
A Bun handler doing `Redis GET → JSON respond` costs ~0.3ms of CPU on a cache hit, ~1.5ms on a miss-then-DB. So:

| RPS per pod | All cache hits | 10% miss |
|---|---|---|
| 100 | 30m CPU | 45m CPU |
| 500 | 150m CPU | 225m CPU |

If CPU request = 300m, then 500 RPS/pod hits at ~50–75% of request. Below 70% threshold (or borderline) — exactly the "CPU isn't moving enough" regime.

### Where does latency come from then?
At 500 concurrent RPS into a single pod with DB pool of 10, the event loop queues. Each request waits ~5–50ms behind others. p99 climbs to 1–3 seconds while CPU sits at 40–50%. This is the textbook "I/O-bound saturation hidden from CPU" pattern.

### Replica math
Pick a target like **100 RPS per pod**:
- Baseline 100 RPS → 1 pod needed, min=2 keeps 2
- Burst 1000 RPS → 1000/100 = 10 pods needed → hits max=10
- Drain → returns to 2

10× scale-up exactly matches the exercise's "10× alert storm". Replicas visibly march 2 → 10 → 2.

### Test duration
HPA defaults: scale-up evaluated ~15s, scale-down has a 5-min stabilization window. So:
- 60s baseline (establish steady state)
- 4 min burst (lets HPA reach max=10)
- 5 min drain (lets scale-down complete back to 2)
- **~10 min total** per scenario

We tune `behavior.scaleDown.stabilizationWindowSeconds: 120` on the HPA to shorten the demo to ~7-8 min while still being principled.

## Concrete proposal

| Knob | Value | Why |
|---|---|---|
| CPU request | **300m** | Big enough that cache-hit traffic stays at 40–60% utilization at burst |
| CPU limit | **1000m** | Headroom — we don't want CPU throttling to confound the latency narrative |
| Memory request / limit | **128Mi / 256Mi** | Bun + small caches + connection pools fit easily |
| Min / max replicas | **2 / 10** | Min=2 satisfies PDB minAvailable=2; max=10 fits 10× burst |
| HPA "bad" signal | **CPU 70% of request** | The exercise's stated baseline |
| HPA "good" signal | **RPS = 100 / pod** | Tied to measured saturation point; easy to explain |
| Baseline load | **100 RPS** | 50/pod with min=2 — quiet, well within budget |
| Burst load | **1000 RPS** | 10× spike (matches spec). At min=2 = 500/pod = saturates event loop |
| Burst duration | **4 min** | Long enough to clear HPA evaluation and reach max |
| Drain duration | **5 min** | With stabilization-window-seconds=120, returns to min=2 |
| DB pool per pod | **10** | Small enough to be a meaningful bottleneck on the miss path |
| Cache TTL | **5 min** | Long enough that hot keys stay hot through a burst |
| Seeded IOCs | **10k** | ~100 "hot" (90% of queries), ~9.9k "cold" (10%, exercises miss path) |
